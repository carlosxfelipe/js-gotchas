# Capítulo 6 — Precisão Numérica (A Falha do IEEE 754)

Quando lidamos com números com casas decimais em JavaScript, nos deparamos rapidamente com o meme mais famoso e compartilhado por desenvolvedores da linguagem, que quebra todas as regras da matemática fundamental que aprendemos na escola:

```js
0.1 + 0.2; // 0.30000000000000004
```

Se você trabalha num e-commerce e testa sua aplicação no console do navegador, o assombro é imediato: "Como assim o total do carrinho de 30 centavos virou essa dízima gigante?"

### O Padrão IEEE 754

Isso não é um bug exclusivo ou erro de projeto do JavaScript. Virtualmente **todas** as linguagens modernas (Python, Ruby, Java, C++) possuem comportamentos similares sob certas configurações.

Isso acontece porque as linguagens adotam o padrão internacional **IEEE 754** para representar números de "ponto flutuante" via hardware de 64 bits.

Trocando em miúdos para humanos não-robôs: os compiladores enxergam apenas linguagem binária (zeros e uns). Tentar guardar décimos exatos estruturados na base decimal (como o simples `0.1`) é matematicamente insuportável no escopo de potências de base binária 2. O número acaba virando uma "dízima binária periódica" sem fim.  
Como a memória computacional para a variável é limitada, o processador decide cordialmente **cortar** o número num tamanho final, resultando não num `0.10` puro, e sim numa minúscula distorção microscópica que a soma fatalmente acumulará numa cauda visível como `...000000004`!

### A Solução Matemática: `Number.EPSILON`

Sabendo que essa distorção ínfima existe, os criadores do ECMAScript inseriram na linguagem um avaliador especial tolerante: a constante `Number.EPSILON`. Ela representa a menor margem de erro possível mapeada pelo hardware!

Se um dia você realmente precisar testar a soma de casas decimais cruas avaliando por igualdade booleana estrita `===`, afaste-se da comparação de praxe e agrupe a lógica pela "ausência de margem de tolerância" através da constante Épsilon nativa, comparando com o valor absoluto (`Math.abs`):

```js
const a = 0.1;
const b = 0.2;

// O teste desastroso:
a + b === 0.3; // false

// O teste maduro e profissional:
Math.abs(a + b - 0.3) < Number.EPSILON; // true!
```

Ou, criando uma função reutilizável para ficar mais claro:

```js
console.log("\n--- A SOLUÇÃO COM Number.EPSILON ---");
// O Number.EPSILON é um número minúsculo (cerca de 2.22e-16).
// Ele representa a menor diferença matemática significativa no JavaScript.

function margemDeErroAceitavel(numero1, numero2) {
  // A função Math.abs() pega o valor absoluto (sempre positivo).
  // Verificamos se a diferença entre os números é menor que a "margem de erro" do JavaScript.
  let diferenca = Math.abs(numero1 - numero2);
  return diferenca < Number.EPSILON;
}

let resultadoDaSoma = 0.1 + 0.2;
let resultadoEsperado = 0.3;

let ehIgual = margemDeErroAceitavel(resultadoDaSoma, resultadoEsperado);

console.log(
  `A diferença entre ${resultadoDaSoma} e ${resultadoEsperado} é menor que Number.EPSILON? ->`,
  ehIgual,
); // true!
```

### Lidando Com Moeda na Prática da Aplicação (Centavos)

Ter que espalhar funções de validação com `Math.abs` e `Number.EPSILON` por todo o sistema não é escalável, tornando o código muito mais complexo de ler e manter por toda a equipe. Como lidar com valores financeiros de forma simples e segura?

A abordagem soberana global do mercado inteiro é **eliminar os pontos flutuantes do tráfego do servidor.**

Você não trabalha com decimais e simuladores de "reais e centavos quebrados". Multiplique todo valor originário submetido pela sua interface vezes o montante redondo para transformar os números em sub-valores isolados, processados rigorosamente no Back-End como Integer, que não sofrem do problema do IEEE 754.

Geralmente, trabalha-se apenas na casa dos centavos cheios (`* 100`), multiplicando os valores para torná-los inteiros (e depois dividindo na hora de exibir):

```js
// O problema clássico resolvido com o truque da multiplicação:
const somaSegura = (0.1 * 100 + 0.2 * 100) / 100;
console.log(somaSegura); // 0.3 exato!

// Na prática em sistemas de e-commerce e bancos:
// Você precisa processar R$ 1,50.
// Trate esse valor como 150 centavos numéricos.

const precoAgua = 150; // Centavos puros sem decimais
const precoCafe = 525; // Representação de R$ 5,25

const totalReal = precoAgua + precoCafe; // 675. Zero risco de 0.0004 surgir!
```

Para que isso seja seguro de ponta a ponta, o Padrão Ouro da indústria dita que fluxos financeiros inteiros ajam da seguinte forma:

1. **O Backend guarda apenas inteiros:** No Banco de Dados, um preço nunca é gravado como `10.50`, e sim como `1050`. O banco e as APIs enxergam apenas números puros redondos.
2. **A Comunicação (API) não trafega decimais:** O dado empacotado que o App baixa chega como `{"preco": 1050}`. Não existem vírgulas correndo risco de conversão incorreta na internet.
3. **O Frontend formata no último segundo:** Sua UI recebe o número de centavos, faz qualquer cálculo de carrinho somando esses inteiros precisos e, unicamente no instante de pintar a tela para o cliente, divide o resumo por `100` convertendo em texto final seguro com as devidas APIs visuais (como `Intl.NumberFormat`).

### Limites de Representação: `MAX_SAFE_INTEGER`

Da mesma forma que as decimais batem na casa do limite imposto pelos bytes empilhados, o mundo dos inteiros tem seu topo seguro absoluto, exposto na barreira:

```js
Number.MAX_SAFE_INTEGER; // 9007199254740991
```

O JS aceita que, uma vez passado esse abismo de nove quatrilhões, números gigantes somados pela máquina acabarão corrompidos (onde `9007199254740991 + 1` pode estranhamente ser igual a ele mesmo mais `2` sem apitar falha). Caso sua API lide com Hashs massivos ou simulação universal de estrelas e astronomia, a indicação para a nova era é pular de cara a tipagem comum _Number_ e recorrer aos instanciadores uniparadigmáticos estritos conhecidos como `BigInt` (representados por `10000000000n` ou o construtor global explícito `BigInt()`).
